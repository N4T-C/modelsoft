Functional vs Non-Functional Requirements

Trick to remember: Functional = WHAT the system does. Non-Functional = HOW WELL it does it (speed, availability, constraints).

FR Functional Requirements

These are features/actions the system must perform:

1.Withdraw cash in multiples of 100
2.Deposit cash in multiples of 100
3.Balance enquiry
4.Transfer amount between two accounts
5.Print receipt
6.Validate PIN (3 attempts max)
7.Retain card on 3 failed PIN attempts
8.Record all transactions per account
9.Operator can start/stop machine
10.Operator enters total cash before starting

NFR Non-Functional Requirements

These are constraints/quality attributes:

1.Each transaction must complete within 60 seconds
2.Card retained if machine unused for 2 minutes
3.Available 24×7 (availability)
4.Serves one customer at a time
5.Magnetic stripe card reader (hardware)
6.Keyboard + display for interaction
7.Envelope deposit slot
8.Cash dispenser
9.Receipt printer
10.Operator key switch to start/stop
11.Customer can do multiple transactions in one session

Use Case Descriptions (for written exam)

UC1Withdraw Cash — Customer inserts card → enters PIN → selects withdraw → enters amount (multiple of 100) → cash dispensed → receipt printed
UC2Validate PIN — System reads card → asks for PIN → checks with bank → allows 3 attempts → retains card on 3 failures
UC3Balance Enquiry — After auth → customer selects balance → system queries bank → displays balance on screen
UC4Transfer — Customer selects transfer → enters source + dest account + amount → bank processes → confirmation shown
UC5Operator Start/Stop — Operator turns key → enters total cash on hand → machine starts/stops servicing customers

ER Diagram (Entity-Relationship)

ER Diagram shows the data model. Rectangles = Entities, Ovals = Attributes, Diamonds = Relationships, Lines show cardinality (1, M, N).

Key relationships: Customer owns many Cards (1:M). Card linked to Account (1:1). Customer makes many Transactions (1:M). Transaction done via ATM Machine.

Common Exam Questions & Answers

Q1What is DFD? — Data Flow Diagram. Shows how data moves through a system. Has 4 symbols: External Entity (rectangle), Process (circle), Data Store (open rect), Data Flow (arrow).
Q2Difference: FR vs NFR? — FR describes what system does (functions). NFR describes how well it does it (performance, reliability, availability).
Q3What is a Use Case? — A description of how a user (actor) interacts with the system to achieve a goal. Shows actors + system boundary + use cases (ovals).
Q4What is ER Diagram? — Entity Relationship diagram. Models the database. Shows entities (rectangles), attributes (ovals), relationships (diamonds), cardinality (1, M, N).
Q5What is Context Diagram? — Level 0 DFD. Shows entire system as ONE process. Shows all external entities and data flows to/from system. No internal details.
Q6What are the ATM transaction types? — Withdraw, Deposit, Balance Enquiry, Transfer, Print Receipt. All must complete within 60 seconds.

Quick Cheat Sheet — What to draw on paper

Context Diagram

1 big circle in middle = ATM. Boxes around it = Customer, Bank, Operator. Arrows = data flows.

Level 1 DFD

Break ATM into processes (P1–P6). Add data stores D1 (Transaction Log), D2 (Account Data).

Use Case

Rectangle box = system. Stick figures outside = actors. Ovals inside = use cases. Lines connect them.

ER Diagram

Entities: Customer, Card, Account, Transaction, ATM. Show 1:M, 1:1 cardinalities.

SafeHome DFD (from your document — also likely exam topic)

The SafeHome security system follows the same DFD principles:

  • External Entities: Homeowner, Monitoring Service, Sensors
  • Main Processes: Configure System, Monitor Sensors, Detect Event, Dial Phone, Display Status
  • Data Flows: Sensor data → Monitor process → trigger alarm/call
  • Data Stores: Configuration data (passwords, phone numbers, sensor types)
  • Key NFR: Redials every 20 seconds until connection obtained
